# Task: Implement attendance API with cutoff and concurrency safety

## Description

Build the attendance endpoints with Zod validation, enforce the 2-hour signup deadline, protect against race conditions with atomic transactions, and add rate limiting on the signup endpoint.

## Acceptance Criteria

- [ ] `POST /api/games/:id/attend` requires JWT auth and validates body `{ status: 'confirmed' | 'declined' }` via Zod
- [ ] Rejects requests if current time is within 2 hours of `games.scheduled_at` with a clear error message
- [ ] Atomic transaction wraps cutoff check and `INSERT ... ON CONFLICT (game_id, user_id) DO UPDATE`
- [ ] Rate limiter allows max 10 requests per minute per IP/user on this endpoint
- [ ] `GET /api/games/:id/attendance` requires JWT auth and returns `{ confirmed: [...], declined: [...], pending: [...] }`
- [ ] `pending` is computed as all authenticated users minus confirmed/declined (or use a defined team roster if available)
- [ ] Integration tests cover: successful signup, cutoff rejection, concurrent signups, invalid status, and rate limiting

## Technical Details

- Use `pg.Pool` transactions (`BEGIN ... COMMIT`) for atomicity
- Apply `express-rate-limit` or existing project rate-limiting middleware
- Return standard API envelope on all responses
- Error messages must not leak internal details
